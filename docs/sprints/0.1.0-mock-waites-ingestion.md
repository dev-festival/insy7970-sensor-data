# Sprint 0.1.0 - Mock Waites Ingestion

## Problem Statement

The project needs a trustworthy data contract before live Waites source API calls are introduced. This sprint builds a mock Waites ingestion path that behaves like the real pipeline: request parameters go in, raw endpoint responses are preserved as evidence, and lightweight metadata records what happened. The FastAPI service should be able to report what raw evidence is available.

## Requirements

1. Add mock fixtures for Waites `equipment`, `installation-points`, `readings/rms`, `readings/impact-vue`, `readings/temperature`, and `action-items`.
2. Implement a Waites client boundary that can build endpoint names, request parameters, and response envelopes without hardcoding secrets.
3. Add `sensor-data waites fetch --source mock --date YYYY-MM-DD --facility 679` to save raw JSON files under `data/raw/waites/date=YYYY-MM-DD/`.
4. Write a manifest beside the raw files containing source mode, endpoint, request parameters, fetched timestamp, output path, and record count.
5. Add small processed reference outputs for equipment and installation points under `data/processed/waites/reference/`.
6. Add a read-only FastAPI endpoint that lists available raw Waites mock runs and their manifest summaries.
7. Document the mock ingestion command and expected output paths in the README.

## In Scope

- Mock data fixture format.
- Raw evidence file naming.
- Waites endpoint/request parameter construction.
- Manifest writing.
- CLI command for mock fetches.
- Read-only FastAPI status endpoint for raw Waites evidence.
- Processed equipment and installation point reference files.

## Out of Scope

- Network calls to the Waites API.
- Maximo integration.
- Snapshot aggregation from readings.
- Clustering and drift calculations.
- Browser UI beyond the existing static shell.
- API routes that serve processed snapshots or trends.
- Scheduling or background jobs.

## Test Out Plan

- Run `uv run pytest` with fixture-backed tests for each mock endpoint.
- Test that `sensor-data waites fetch --source mock` writes one raw JSON file per endpoint into a temporary data directory.
- Test that manifests include endpoint names, facility ID, selected date, record counts, and paths.
- Test the FastAPI raw-run endpoint with `TestClient`.
- Test that no test requires a Waites API token or network access.
- Test that processed equipment and installation point reference outputs preserve `equipment_id`, `installation_point_id`, `sensor_id`, and `customer_asset_id`.

## Next Sprint Lead In

Sprint `0.2.0` consumes the raw mock readings and reference files to produce daily sensor snapshots and simplified trends. After that sprint, the project should be useful without live credentials.
