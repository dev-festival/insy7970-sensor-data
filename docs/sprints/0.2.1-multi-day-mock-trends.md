# Sprint 0.2.1 - Multi-Day Mock Trends

## Problem Statement

The project can build trend-ready files, but the current mock dataset only proves the file contract. This sprint adds enough controlled multi-day mock variation to make trends visible, explainable, and testable before clustering depends on those processed outputs.

## Requirements

1. Add a small multi-day mock fixture strategy that can generate or select at least three source dates without requiring live Waites access.
2. Include deliberate trend patterns: one stable sensor, one rising vibration sensor, one falling or normalizing sensor, one temperature spike, and one missing-day case.
3. Preserve the existing raw artifact contract under `data/raw/waites/date=YYYY-MM-DD/` for every mock date.
4. Ensure `snapshot build` works unchanged against each mock date and produces comparable daily rows.
5. Ensure `trend build` over the mock date range produces visibly different values across dates for selected metrics.
6. Document the mock dates, intended sensor behaviors, and recommended example commands.

## In Scope

- Multi-day mock source data for Waites reading endpoints.
- Small fixture helpers or fixture directories if they keep the raw contract clear.
- Trend examples that prove date-to-date movement.
- Tests for missing-day behavior and stable output paths.
- README or CLI doc updates for the multi-day mock trend workflow.

## Out of Scope

- Real Waites API calls.
- Maximo/DB2 data.
- Clustering or drift modeling.
- Browser chart polish.
- Synthetic data generators that obscure the fixture values.
- Large fixture datasets.

## Test Out Plan

- Run `uv run pytest`.
- Test that mock fetch can write raw evidence for every supported mock trend date.
- Test that snapshot artifacts are written for each generated mock date.
- Test that `trend build` over the date range writes `sensor_trends.csv`, `equipment_trends.csv`, and `metadata.json`.
- Test one stable sensor remains nearly flat across dates.
- Test one rising vibration sensor has increasing values across dates.
- Test missing snapshot dates are reported in `skipped_dates` without crashing when at least one date exists.

## Next Sprint Lead In

Sprint `0.2.2` introduces a narrow live Waites canary. The multi-day mock trend data from this sprint remains the primary behavior testbed while live data validates API assumptions at the boundary.
