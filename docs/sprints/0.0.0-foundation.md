# Sprint 0.0.0 - Project and Service Foundation

## Problem Statement

The project needs a clean, runnable uv foundation before sensor logic lands. A stranger should be able to clone the repo, install dependencies, run CLI and FastAPI health checks, and run tests without access to Waites, Maximo, ODBC drivers, or plant network resources.

## Requirements

1. Define a package under `src/` with clear top-level boundaries for config, storage paths, CLI commands, FastAPI service code, static web assets, Waites integration, Maximo integration, snapshots, and clustering.
2. Configure `pyproject.toml` with project metadata, runtime dependencies, dev dependencies, and a console script such as `sensor-data`.
3. Add `.env.example` with placeholder values for Waites API configuration, Maximo ODBC configuration, app mode, and data directory settings.
4. Update `.gitignore` so `.env`, `.venv/`, `data/`, `app.db`, caches, build output, and Python bytecode do not enter Git.
5. Add a Typer CLI skeleton with `sensor-data --help`, `sensor-data health`, and `sensor-data serve --source mock` working through `uv run`.
6. Add a minimal FastAPI app with `/health`, route registration structure, settings wiring, and static-file mount points for the browser app.
7. Add a minimal pytest harness for CLI and FastAPI health, and fill the README with install steps, environment setup, service startup, the selected source API docs placeholder, and the one-line build objective.

## In Scope

- Project layout and naming.
- CLI entry point, health command, and serve command shell.
- FastAPI app factory, `/health`, and static app shell.
- Config loading from environment variables.
- Data path helpers for `data/raw/` and `data/processed/`.
- First pytest files and test configuration.
- Basic README and contributor quickstart.

## Out of Scope

- Real Waites API requests.
- Real Maximo DB2/ODBC connection work.
- Data processing, clustering, or trend calculations.
- Data-serving FastAPI routes beyond health and static shell.
- Full dashboard UI.
- User profile persistence.

## Test Out Plan

- Run `uv run sensor-data --help` and confirm the CLI starts.
- Run `uv run sensor-data health` and confirm it returns stable machine-readable output.
- Run `uv run sensor-data serve --help` and confirm service startup options are discoverable.
- Run `uv run pytest` and include tests for config defaults, `.env` override behavior, path creation using `tmp_path`, CLI execution using Typer's test runner, and `/health` using FastAPI's `TestClient`.
- Run `git diff --check` before committing to catch whitespace issues.
