# Sprint 0.3.0 - Clustering

## Problem Statement

The project needs to group sensors by behavior so reliability users can spot similar operating patterns and outliers. This sprint adds deterministic clustering over processed sensor snapshots while keeping the implementation callable from the CLI and testable with mock data.

## Requirements

1. Select numeric feature columns from processed snapshots while excluding identifiers, names, asset numbers, and non-feature metadata.
2. Normalize features before clustering using a reusable scaler step.
3. Run KMeans with a fixed random seed and configurable cluster count.
4. Save clustered sensor rows, cluster summary statistics, clustering metrics, and optional PCA coordinates under `data/processed/clusters/`.
5. Add `sensor-data cluster run --date YYYY-MM-DD --k 4 --source mock`.
6. Add a basic cluster drift artifact that compares cluster assignments or centroids between two processed dates.
7. Document how to choose `k` and where cluster outputs are written.

## In Scope

- Feature selection.
- Scaling, KMeans, PCA coordinates, and cluster summaries.
- Silhouette and Calinski-Harabasz metrics where data size permits.
- CLI output for cluster runs.
- First drift comparison artifact.
- Pytest coverage over deterministic mock inputs.

## Out of Scope

- Production-grade anomaly detection.
- Interactive visual tuning of clusters.
- Model registry or persisted fitted model storage.
- Real-time clustering.
- Maximo overlay logic.
- Web visualization.

## Test Out Plan

- Run `uv run pytest` with a fixed mock snapshot and deterministic clustering settings.
- Test feature selection excludes identifier and metadata fields.
- Test output files include `cluster`, metric values, and cluster summary rows.
- Test metrics are skipped or marked unavailable when sample size is too small.
- Test drift comparison produces stable output for two known fixture snapshots.
- Test CLI command writes files into a temporary data directory.
