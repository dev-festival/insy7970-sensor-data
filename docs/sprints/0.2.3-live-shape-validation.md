# Sprint 0.2.3 - Live Shape Validation

## Problem Statement

After live raw capture exists, the project needs to prove that live Waites responses can enter the same processing chain as mock evidence. This sprint validates real response shapes, records any contract differences, and keeps the snapshot and trend builders stable before clustering starts using their outputs.

## Requirements

1. Add validation checks for required Waites endpoint envelopes and fields before snapshot processing begins.
2. Add `waites validate --source mock|api --date YYYY-MM-DD` and write `validation.json` beside the raw run.
3. Compare live raw files against the mock data contract and report missing fields, unexpected envelope shapes, null-heavy columns, endpoint-level record counts, source mismatches, and manifest fetch errors.
4. Allow `snapshot build --source api` to read live raw artifacts once validation passes.
5. Allow `trend build --source api` only for existing live snapshots, while keeping date ranges explicit and small.
6. Update the mock data contract when live evidence proves a field shape or envelope assumption is wrong.
7. Add CLI output that distinguishes validation warnings from hard failures while keeping normal pytest runs offline and deterministic.

## In Scope

- Raw artifact schema validation.
- Validation report artifacts under `data/raw/waites/date=YYYY-MM-DD/validation.json`.
- Source-aware snapshot and trend building for `mock` and `api`.
- Clear CLI errors for missing or incompatible live files.
- Contract documentation updates based on live evidence.
- Tests using recorded or fake live-shaped fixtures.
- Small manual smoke workflow from live fetch to snapshot to trend.

## Out of Scope

- Large live data performance tuning.
- Full production observability.
- Automatic schema migration.
- KMeans clustering.
- Maximo joins.
- Browser trend visualization.

## Test Out Plan

- Run `uv run pytest`.
- Test validators accept current mock fixtures.
- Test validators catch missing required fields and malformed envelopes.
- Test validators warn, without failing, for tolerated null-heavy fields and newly observed optional fields.
- Test `waites validate` writes a validation report and exits nonzero on hard validation errors.
- Test `snapshot build --source api` against recorded live-shaped fixture data.
- Test `trend build --source api` reads existing API-source snapshots and writes trend artifacts.
- Test warnings are included in metadata without causing crashes for tolerated nulls or optional fields.
- Run a documented manual canary sequence against one real date when credentials and network are available.

## Implementation Notes

Validation is a source-bound gate, not a transformation step. It reads raw files, compares them to the current Waites contract, writes a compact report, and lets downstream tools proceed only when `error_count` is zero. Warnings remain visible in CLI output and snapshot metadata so the analyst can see odd source behavior without losing useful daily processing.
