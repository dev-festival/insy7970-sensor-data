# Sprint 0.5.0 - Maximo Integration

## Problem Statement

Sensor data becomes more valuable when it is aligned with maintenance history. This sprint adds a Maximo data boundary that can use DB2/ODBC in production and mock fixtures in development, joining records to Waites equipment through asset number alignment.

## Requirements

1. Add Maximo configuration for DSN, schema, query timeout, and mock mode without committing credentials.
2. Add a query loader for parameterized SQL files using positional ODBC parameters.
3. Implement a read-only asset maintenance history query by `assetnum`.
4. Add fixture-backed Maximo mock data so tests and local demos do not require DB2 access.
5. Join Maximo asset history to Waites equipment using `customer_asset_id` aligned to Maximo `assetnum`.
6. Add `sensor-data maximo asset-history --assetnum VALUE --source mock`.
7. Expose maintenance history through the API for equipment and sensor detail views.

## In Scope

- Read-only DB2/ODBC access boundary.
- Query file organization.
- Mock Maximo fixtures.
- Asset history CLI command.
- Core join logic between Waites and Maximo data.
- API endpoint for maintenance history lookup.

## Out of Scope

- Writing to Maximo.
- Full CMMS replication.
- User authentication.
- Report scheduling.
- Advanced work order forecasting.
- Production deployment.

## Test Out Plan

- Run `uv run pytest` with mock Maximo fixtures.
- Test SQL query loading and missing-query behavior.
- Test ODBC parameter ordering without requiring a live DB2 connection.
- Test asset alignment joins preserve unmatched Waites equipment and unmatched Maximo records where appropriate.
- Test CLI asset-history command returns stable fixture-backed output.
- Test API maintenance endpoint returns useful empty results for unknown assets.
