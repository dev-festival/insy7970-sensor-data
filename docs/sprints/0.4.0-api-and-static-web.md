# Sprint 0.4.0 - API and Static Web Hardening

## Problem Statement

The project already has a FastAPI service and static app shell by this point. This sprint hardens the browser-accessible surface for daily review, adds richer API responses for clustering and drift, and improves the static web app without moving business logic into route handlers or browser JavaScript.

## Requirements

1. Expand FastAPI endpoints for equipment lookup, cluster results, drift results, and richer snapshot detail responses.
2. Preserve existing health, raw-run, snapshot, and trend endpoints while tightening response schemas and error handling.
3. Serve the static HTML/CSS/JS app from the API process for local development and mock review.
4. Render trend, snapshot, cluster, and drift views using Plotly.js against API responses.
5. Keep `sensor-data serve --source mock` as the single local service startup command.
6. Keep route handlers thin by delegating loading, filtering, and calculations to core modules.
7. Document local API and web startup in the README.

## In Scope

- FastAPI response schema hardening.
- Read-only mock-data API endpoints for all processed artifacts.
- Static app workflow beyond the shell.
- Plotly-based trend and cluster views.
- Basic browser-friendly error messages for missing data.
- API tests with FastAPI's test client.

## Out of Scope

- Authentication and user profiles.
- Deployment hardening.
- Real Waites API calls from the web request path.
- Maximo ODBC connection.
- Background refresh jobs.
- Complex dashboard customization.

## Test Out Plan

- Run `uv run pytest` with FastAPI `TestClient` tests for health, dates, snapshots, trends, and clusters.
- Test API responses are JSON serializable and stable for fixture data.
- Test missing processed files return useful non-500 responses.
- Test `sensor-data serve --help` starts from the CLI layer.
- Add a small static smoke test that checks expected HTML element IDs or asset paths exist.
