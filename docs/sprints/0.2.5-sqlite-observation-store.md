# Sprint 0.2.5 - SQLite Observation Store

## Problem Statement

The project should not repeatedly load large raw JSON payloads for routine snapshot, trend, and clustering workflows. At the same time, it should not over-aggregate away native timestamps. This sprint creates the "beam" side of the hewing machine: validated native observations are loaded into SQLite with enough indexing and provenance to support daily analyst outputs without keeping raw JSON in memory.

## Requirements

1. Add a SQLite database under the configured data directory, for example `data/processed/observations.sqlite`.
2. Add `sensor-data store load-waites --source api --date YYYY-MM-DD` to load validated raw Waites artifacts into normalized SQLite tables.
3. Preserve native timestamps for RMS, temperature, ImpactVue, equipment, installation points, and action items rather than forcing a single sampling grid.
4. Add indexes for `source_date`, `installation_point_id`, `equipment_id`, `axis`, `timestamp`, and `customer_asset_id` where applicable.
5. Store load metadata including source date, source mode, manifest checksum, row counts, load time, and schema version.
6. Make the load command idempotent for a date, either by replace-on-date or by explicit `--replace`.
7. Update snapshot and trend builders to read from SQLite when requested while preserving existing file-backed mock behavior.

## In Scope

- SQLite schema and migration/version metadata.
- Native observation tables for Waites measurements and reference entities.
- CLI load command with concise JSON summary output.
- Query helpers for daily min, max, mean, sample count, first timestamp, and last timestamp.
- Tests with fixture-backed small datasets.
- Documentation explaining raw evidence versus SQLite observations versus daily snapshots.

## Initial Schema Sketch

This schema is intentionally boring and native-timestamp-first. It should be implemented with the Python standard library `sqlite3` unless a later sprint proves an ORM is worth the weight.

```sql
CREATE TABLE observation_schema (
    version INTEGER PRIMARY KEY,
    applied_at TEXT NOT NULL
);

CREATE TABLE waites_loads (
    source_date TEXT NOT NULL,
    source TEXT NOT NULL,
    facility_id INTEGER NOT NULL,
    manifest_path TEXT NOT NULL,
    manifest_sha256 TEXT NOT NULL,
    loaded_at TEXT NOT NULL,
    schema_version INTEGER NOT NULL,
    equipment_count INTEGER NOT NULL,
    installation_point_count INTEGER NOT NULL,
    rms_count INTEGER NOT NULL,
    impact_count INTEGER NOT NULL,
    temperature_count INTEGER NOT NULL,
    action_item_count INTEGER NOT NULL,
    PRIMARY KEY (source, source_date, facility_id)
);

CREATE TABLE waites_equipment (
    source_date TEXT NOT NULL,
    equipment_id INTEGER NOT NULL,
    asset_tree_id INTEGER,
    name TEXT,
    facility_id INTEGER,
    customer_asset_id TEXT,
    PRIMARY KEY (source_date, equipment_id)
);

CREATE TABLE waites_installation_points (
    source_date TEXT NOT NULL,
    installation_point_id INTEGER NOT NULL,
    name TEXT,
    equipment_id INTEGER,
    sensor_id INTEGER,
    facility_id INTEGER,
    last_seen TEXT,
    installation_date TEXT,
    customer_asset_id TEXT,
    PRIMARY KEY (source_date, installation_point_id)
);

CREATE TABLE waites_rms_observations (
    source_date TEXT NOT NULL,
    timestamp TEXT NOT NULL,
    installation_point_id INTEGER NOT NULL,
    axis TEXT NOT NULL,
    facility_id INTEGER,
    acceleration REAL,
    velocity REAL,
    pk_pk REAL,
    cf REAL,
    PRIMARY KEY (source_date, timestamp, installation_point_id, axis)
);

CREATE TABLE waites_temperature_observations (
    source_date TEXT NOT NULL,
    timestamp TEXT NOT NULL,
    installation_point_id INTEGER NOT NULL,
    facility_id INTEGER,
    value REAL,
    ambient REAL,
    PRIMARY KEY (source_date, timestamp, installation_point_id)
);

CREATE TABLE waites_impact_observations (
    source_date TEXT NOT NULL,
    timestamp TEXT NOT NULL,
    installation_point_id INTEGER NOT NULL,
    axis TEXT,
    facility_id INTEGER,
    impact_vue_acceleration REAL,
    impact_vue_pk_pk REAL,
    PRIMARY KEY (source_date, timestamp, installation_point_id, axis)
);

CREATE TABLE waites_action_items (
    source_date TEXT NOT NULL,
    action_item_id TEXT NOT NULL,
    wo_number TEXT,
    wo_status TEXT,
    sensor_id INTEGER,
    type TEXT,
    status TEXT,
    installation_point_id INTEGER,
    equipment_id INTEGER,
    title TEXT,
    description TEXT,
    urgency TEXT,
    closed_at TEXT,
    facility_id INTEGER,
    raw_json TEXT,
    PRIMARY KEY (source_date, action_item_id)
);

CREATE TABLE waites_daily_metric_rollups (
    source_date TEXT NOT NULL,
    installation_point_id INTEGER NOT NULL,
    equipment_id INTEGER,
    axis TEXT,
    metric TEXT NOT NULL,
    source_table TEXT NOT NULL,
    sample_count INTEGER NOT NULL,
    min_value REAL,
    max_value REAL,
    mean_value REAL,
    first_timestamp TEXT,
    last_timestamp TEXT,
    PRIMARY KEY (source_date, installation_point_id, axis, metric)
);
```

Expected indexes:

```sql
CREATE INDEX idx_waites_rms_sensor_time
    ON waites_rms_observations (installation_point_id, axis, timestamp);

CREATE INDEX idx_waites_temperature_sensor_time
    ON waites_temperature_observations (installation_point_id, timestamp);

CREATE INDEX idx_waites_impact_sensor_time
    ON waites_impact_observations (installation_point_id, axis, timestamp);

CREATE INDEX idx_waites_installation_equipment
    ON waites_installation_points (source_date, equipment_id);

CREATE INDEX idx_waites_equipment_asset
    ON waites_equipment (source_date, customer_asset_id);

CREATE INDEX idx_waites_daily_metric_lookup
    ON waites_daily_metric_rollups (source_date, metric, equipment_id, installation_point_id);
```

The rollup table is derived, not canonical. The native observation tables remain the source for rebuilding daily facts if aggregation rules change.

## Out of Scope

- Maximo SQLite ingestion.
- User profile database design.
- Production database server.
- Heavy ORM adoption.
- Complex resampling or interpolation.
- Browser trend query UI.
- Clustering implementation.

## Test Out Plan

- Run `uv run pytest`.
- Test schema creation from an empty data directory.
- Test loading a validated mock or recorded live-shaped raw date inserts expected row counts.
- Test idempotent reload behavior for the same date.
- Test timestamp-native observations keep RMS hourly-ish samples and temperature 15-minute samples without coercion to a common grid.
- Test daily aggregate query helpers return min, max, mean, sample count, first timestamp, and last timestamp.
- Test snapshot and trend builders can use SQLite-backed observations for one date and a small date range.

## Next Sprint Lead In

Sprint `0.3.0` adds clustering on top of daily snapshot facts. With validated live shapes, raw lifecycle controls, and SQLite-backed observations in place, clustering can focus on feature selection and model behavior instead of raw payload handling.
